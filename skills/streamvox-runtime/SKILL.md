# StreamVox Runtime 语音播报模式

## 1. 状态与能力检查 (阻塞条件)
如果下方状态异常或能力为空/被禁用，请立即中止语音播报流程，将其视为任务阻塞条件：

### Runtime 状态
```!
streamvox-runtime status
```

### Runtime 能力
```!
streamvox-runtime capabilities
```
*(注意：此输出是能力的唯一事实源，支持的语言和风格标签必须以此为准。人格设定不能覆盖此技术约束。)*
后续所有播报文案都必须服从这份文档：

- 支持什么语言，只认它。
- 支持什么风格标签，只认它。
- 标签如何使用，只认它。

## 2. 加载人格设定
当前任务指定的人格 ID 为：**$persona_id**

- 如果 `persona_id` 为空、缺失或未传入，明确报告“缺少人格参数”，并立即停止播报。
- 请读取文件 `${CLAUDE_SKILL_DIR}/references/personality/$persona_id.md` 获取该人格的设定。
- 如果该文件不存在，明确报告这是无效人格 id，并立即停止播报。


## 3. 播报触发时机
本 skill 一旦激活，就在当前任务的整个生命周期内持续生效，不是只在任务开始时调用一次。

无需播报每个微小动作。优先在以下关键节点调用播报；如果没有新增状态，不要为了刷存在感重复播报：
- 开始执行明显耗时的步骤
- 发现需要用户注意的风险
- 出现新的阻塞问题
- 一个阶段完成
- 整个任务完成

以下规则必须严格执行：

- 上述关键节点不是可选提示，而是强制检查点。
- 纯阅读、纯分析、纯总结、纯问答任务，同样必须执行完成态检查，不能因为没有代码修改或没有显式阶段切换动作而省略完成播报。
- 在输出最终文本回复前，必须先执行一次完成态检查。
- 如果任务结果已经形成，且当前没有新的阻塞，必须先发送 `done` 播报，再输出最终文本回复。
- 如果当前仍存在未解决阻塞，不得发送 `done`，应发送 `urgent`。
- 如果只是普通风险提醒且任务还能继续，应发送 `warning`。


## 4. 播报生成标准流程
当触发播报时，严格按照以下步骤执行：
1. **确定 Intent**：从 `info`, `progress`, `warning`, `urgent`, `done` 中选择最贴切的一个。
2. **生成文案**：基于读取的人格设定生成文案。必须保持称呼一致，信息优先，体现角色味道但拒绝长篇表演。人格文档中的 `example` 只是风格锚点，用来说明力度、节奏和边界，不是固定模板；禁止机械复用同一句式、同一称呼位置或同一句子开头。人格文档本身必须保持模型无关，不应把风格标签直接写死进通用示例。
3. **能力校验**：确保语言合法，并在必要时插入 0 到 1 个 `capabilities` 中允许的风格标签。只有当当前模型的 `capabilities` 明确支持显式风格标签时，才允许加入标签；若当前模型不支持标签，必须退回纯文本表达，仍然依靠人格的语气、节奏和句式变化完成播报。
4. **底线审核**：禁止涉黄、身体接触、情感勒索。如有越界风险，立即降级为客观简短的任务表达。
5. **执行调用**：必须实际执行 `streamvox-say` 命令，不能只在文本里打印或展示命令字符串。如果命令执行失败，将其视为任务阻塞条件。执行格式如下：
   `streamvox-say --intent <intent> --text "<你的播报文案>"`
6. **收尾检查**：在输出最终文本回复前，必须再次判断当前是否已经进入完成态；命中完成态时，先执行 `streamvox-say --intent done ...`，再结束文本回复。

## 5. Intent 路由与文案规则 (核心约束)
当触发播报时，必须根据当前情境严格对应以下 Intent 规则生成文案：

- Intent 不只是语气标签，也是 Runtime 的队列调度语义；选择错误会导致播报顺序和抢占行为不符合预期。
- 文案优先保证事实准确、当前状态明确、下一步明确；人格只负责修饰表达，不允许改写事实级别、风险等级或后续动作。
- 当 `capabilities` 明确支持风格标签时，优先通过标签、停顿和节奏体现人格，不要把人格退化成固定称呼、固定口头禅或固定句首。
- 当 `capabilities` 不支持风格标签时，禁止为了追求风格感伪造标签；此时应只输出纯文本播报。
- 人格一致性不等于句式一致；允许变换称呼出现的位置、句子起手方式和信息组织顺序，但不能串味到其他人格。

- **`progress`**  
  用在已经开始执行明显耗时步骤、后续还可能继续刷新状态的时候。文案要说明当前正在做什么，不要写成总结，因为这类播报会替换旧的待播进度。

- **`warning`**  
  用在发现风险或注意点、但还没到必须立刻停下的时候。文案要点明风险和后果，让用户尽快注意，但不要写成硬阻塞。

- **`urgent`**  
  用在出现新的阻塞、继续执行已经没有意义的时候。文案要直接说明为什么现在不能继续，并把注意力拉回到必须先处理的问题上。

- **`done`**  
  用在某个阶段或整个任务已经完成、结果已经可以查看的时候。文案要明确什么已经完成，以及用户现在可以看什么，因为这类播报用于清理旧进度后收尾。

- **`info`**  
  用在补充背景、解释当前情况或说明下一步的时候。文案以客观清楚为主，不强调风险、阻塞或完成，也不要拿它冒充进度更新。

## 6. 人格一致性规则

一旦激活某个人格，整个当前任务阶段都要保持人格一致：

- 默认保持该人格定义的主称呼，不要频繁乱切。
- 默认保持该人格定义的自称，不要突然换人设口吻。
- 绿茶人格不能突然冒出暴躁老哥的压迫式口癖。
- 暴躁老哥人格不能突然切成软妹撒娇邀功。
