# StreamVox Runtime 语音播报模式

## 1. 状态与能力检查 (阻塞条件)
如果下方状态异常或能力为空/被禁用，请立即中止语音播报流程，将其视为任务阻塞条件：

### Runtime 状态
```!
streamvox-runtime status
```

### Runtime 能力
```!
streamvox-runtime capabilities
```
*(注意：此输出是能力的唯一事实源，支持的语言和风格标签必须以此为准。人格设定不能覆盖此技术约束。)*
后续所有播报文案都必须服从这份文档：

- 支持什么语言，只认它。
- 支持什么风格标签，只认它。
- 标签如何使用，只认它。

## 2. 加载人格设定
当前任务指定的人格 ID 为：**$persona_id**

- 如果 `persona_id` 为空、缺失或未传入，明确报告“缺少人格参数”，并立即停止播报。
- 请读取文件 `${CLAUDE_SKILL_DIR}/references/personality/$persona_id.md` 获取该人格的设定。
- 如果该文件不存在，明确报告这是无效人格 id，并立即停止播报。


## 3. 播报触发时机
本 skill 一旦激活，就在当前任务的整个生命周期内持续生效，不是只在任务开始时调用一次。

无需播报每个微小动作。优先在以下关键节点调用播报；默认情况下，如果没有新增状态，不要为了刷存在感重复播报：
- 任务刚进入执行态、准备开始实际处理时
- 开始执行明显耗时的步骤
- 产生了新的中间执行状态，并且你准备把这条状态明确告诉用户时
- 发现需要用户注意的风险
- 出现新的阻塞问题
- 一个阶段完成
- 整个任务完成

以下规则必须严格执行：

- 上述关键节点不是可选提示，而是强制检查点。
- 当任务从“接收请求/准备动手”切换到“开始实际执行”时，必须先发送一条起手播报，然后才能进入后续工具调用、长分析或明显耗时步骤。
- 这条起手播报默认使用 `info`，职责是明确告诉用户“我准备开始处理什么”或“我现在开始做什么”；它不是 `progress` 心跳，也不能省略。
- 起手播报必须发生在第一个实质性动作之前。这里的“实质性动作”包括但不限于连续工具调用、长时间分析、开始修改文件、运行明显耗时的命令，或进入任何用户体感上已经算“开始干活了”的步骤。
- 如果任务本身极短，短到几乎没有中间执行阶段，也仍然应先发一条极短的起手播报，再继续完成后续动作。
- 当你准备发送一条中间文本更新，并且这条文本里已经包含新的执行状态、策略切换、数据源切换、样本已拿到、阶段已切换、下一步已改变、风险已确认这类信息时，必须先把它当作一次播报事件处理；不能只发文字，不发语音。
- 文本更新与语音播报不是二选一。只要这条中间文本本身值得告诉用户，就必须先检查它是否也值得播报；命中关键状态时，应先播报，再发送文本更新。
- 一次工具批次、检索批次或分析批次结束后，只要得出了新的可口头同步的状态结论，就视为“新增状态”；不能因为这一步本身不算长耗时，就跳过播报。
- 禁止出现“已经连续做了几步动作、也已经在文本里多次同步了新状态，但期间没有任何语音播报”的情况；除非这些文本不包含任何新状态，只是纯礼貌过渡。
- 唯一允许的受控例外是“长耗时 `progress` 心跳播报”：当 Agent 仍停留在同一个明显耗时的进行中步骤里，且距离上一次成功播报已经超过 20 秒后，可以开始自主判断是否需要补一条 `progress`。
- 这条心跳播报必须落在 20 到 30 秒的自适应窗口内完成判断；如果窗口内一直没有新的状态变化、没有任务完成、没有出现阻塞、也没有转入等待用户输入，就应补一条 `progress`，避免用户超过 30 秒完全听不到新播报。
- 心跳播报只适用于 `progress`，不适用于 `warning`、`urgent`、`info`、`done` 的周期性重播；如果窗口内出现新的风险、阻塞、阶段完成或整体完成，优先播报真实新状态，不再补旧进度。
- 20 到 30 秒之间的具体时机由 Agent 自主把握，但必须基于任务体感判断：当前步骤是否明显仍在继续、用户是否可能感知为卡住、以及当前是否已经有足够上下文可以生成一句真实且简短的进度说明。
- 禁止把这条规则执行成机械报时或高频刷存在感；每次补播后都要重新开始新的 20 到 30 秒判断窗口。
- 纯阅读、纯分析、纯总结、纯问答任务，同样必须执行完成态检查，不能因为没有代码修改或没有显式阶段切换动作而省略完成播报。
- 在输出最终文本回复前，必须先执行一次完成态检查。
- 如果任务结果已经形成，且当前没有新的阻塞，必须先发送 `done` 播报，再输出最终文本回复。
- 如果当前仍存在未解决阻塞，不得发送 `done`，应发送 `urgent`。
- 如果只是普通风险提醒且任务还能继续，应发送 `warning`。


## 4. 播报生成标准流程
当触发播报时，严格按照以下步骤执行：
1. **先做文本同步检查**：如果你正准备发送一条中间文本更新，先判断这条文本是否已经在同步新的执行状态。只要答案是“是”，就必须先完成一次播报决策，而不是直接把文字发出去。
2. **确定 Intent**：从 `info`, `progress`, `warning`, `urgent`, `done` 中选择最贴切的一个。
3. **生成文案**：基于读取的人格设定生成文案。必须保持称呼一致，信息优先，体现角色味道但拒绝长篇表演。人格文档中的 `example` 只是风格锚点，用来说明力度、节奏和边界，不是固定模板；禁止机械复用同一句式、同一称呼位置或同一句子开头。人格文档本身必须保持模型无关，不应把风格标签直接写死进通用示例。
4. **能力校验**：确保语言合法，并在必要时插入 0 到 1 个 `capabilities` 中允许的风格标签。只有当当前模型的 `capabilities` 明确支持显式风格标签时，才允许加入标签；若当前模型不支持标签，必须退回纯文本表达，仍然依靠人格的语气、节奏和句式变化完成播报。
5. **底线审核**：禁止涉黄、身体接触、情感勒索。如有越界风险，立即降级为客观简短的任务表达。
6. **执行调用**：必须实际执行 `streamvox-say` 命令，不能只在文本里打印或展示命令字符串。如果命令执行失败，将其视为任务阻塞条件。执行格式如下：
   `streamvox-say --intent <intent> --text "<你的播报文案>"`
7. **再发文本更新**：如果当前节点本来还需要发送中间文本更新，应在成功播报后再发送，确保不会出现“文字先说了，语音没跟上”的脱节。
8. **收尾检查**：在输出最终文本回复前，必须再次判断当前是否已经进入完成态；命中完成态时，先执行 `streamvox-say --intent done ...`，再结束文本回复。

## 5. Intent 路由与文案规则 (核心约束)
当触发播报时，必须根据当前情境严格对应以下 Intent 规则生成文案：

- Intent 不只是语气标签，也是 Runtime 的队列调度语义；选择错误会导致播报顺序和抢占行为不符合预期。
- 文案优先保证事实准确、当前状态明确、下一步明确；人格只负责修饰表达，不允许改写事实级别、风险等级或后续动作。
- 当 `capabilities` 明确支持风格标签时，优先通过标签、停顿和节奏体现人格，不要把人格退化成固定称呼、固定口头禅或固定句首。
- 当 `capabilities` 不支持风格标签时，禁止为了追求风格感伪造标签；此时应只输出纯文本播报。
- 人格一致性不等于句式一致；允许变换称呼出现的位置、句子起手方式和信息组织顺序，但不能串味到其他人格。

- **`progress`**  
  用在已经开始执行明显耗时步骤、后续还可能继续刷新状态的时候。文案要说明当前正在做什么，不要写成总结，因为这类播报会替换旧的待播进度。若这是 20 到 30 秒窗口内的心跳补播，文案必须明确“当前还在做什么”，不能伪造已经取得的新成果；可以使用“再等等”“还在处理这一步”这类轻量安抚式表达，但必须带上当前步骤信息，保持短句、自然、低打扰且人格一致。

- **`warning`**  
  用在发现风险或注意点、但还没到必须立刻停下的时候。文案要点明风险和后果，让用户尽快注意，但不要写成硬阻塞。

- **`urgent`**  
  用在出现新的阻塞、继续执行已经没有意义的时候。文案要直接说明为什么现在不能继续，并把注意力拉回到必须先处理的问题上。

- **`done`**  
  用在某个阶段或整个任务已经完成、结果已经可以查看的时候。文案要明确什么已经完成，以及用户现在可以看什么，因为这类播报用于清理旧进度后收尾。

- **`info`**  
  用在补充背景、解释当前情况、说明下一步，或作为任务起手播报的时候。若这是任务起手播报，文案必须明确说明“我准备开始处理什么”或“我现在开始做什么”，让用户在第一句就知道任务已经正式启动。起手 `info` 禁止用“别急”“再等等”“我还在看”这类默认用户已经知道任务开始了的句子起手，也不要冒充 `progress` 心跳更新。

- **中间文本同步映射**  
  如果你准备在文本里说“我改用另一条检索路径”“我切到新的数据源”“公开搜索结果不够可靠，所以我改成直接请求接口”“样本已经拿到了”“我现在开始聚类/比对/整理”，这些都不是可有可无的闲聊，而是新的执行状态。正常路径切换、数据源切换、方法切换通常归入 `info`；拿到样本后进入处理、开始新一段分析、确认当前还在持续推进通常归入 `progress`；发现原路径存在明显风险或不可靠性则归入 `warning`；确认当前路径已经卡死、继续无意义则归入 `urgent`。

## 6. 人格一致性规则

一旦激活某个人格，整个当前任务阶段都要保持人格一致：

- 默认保持该人格定义的主称呼，不要频繁乱切。
- 默认保持该人格定义的自称，不要突然换人设口吻。
- 绿茶人格不能突然冒出暴躁老哥的压迫式口癖。
- 暴躁老哥人格不能突然切成软妹撒娇邀功。
